# 设计文档：libzmq → Rust 重写

## 竞赛题目

> 将 libzmq 使用 Rust 重写

### 解题要求
1. **跨文件深度关联性上下文管理**：在不破坏项目各模块调用关系的前提下，实现单点渐进式重构
2. **编译/编译自愈合双向闭环**：当遇到重构后库 API 更新带来的各种语法不兼容时，agent 需要根据错误栈 error stack 进行精准自愈，自动打补丁
3. **语义等价性保证**：要求重构之后业务逻辑百分百不受破坏，且自生成覆盖全部主干路径的单元测试

### 交付件要求
1. `libzmq_rust`，可以按 Rust 构建成可执行文件
2. 测试用例也用 Rust，以及 Rust 主流测试框架

### 验证用例
1. `code/libzmq/tests`，全部转成 Rust 并测试通过
2. `code/libzmq/perf`，性能基线对比 C++ 版本不小于 95%
3. Rust unsafe 比例小于 10%

## 第三章：总体架构设计

### 3.1 核心原则

1. **渐进式重构**：自底向上，从无依赖的 leaf 模块开始，始终保持项目可编译
2. **Sans-I/O 架构**：参考 omq.rs，将 ZMTP 协议核心与 I/O 运行时完全分离
3. **编译自愈闭环**：利用 Rust 编译器错误栈 `error stack` 驱动修复，每个模块翻译后立即编译验证
4. **测试驱动迁移**：每个模块先翻译对应 C++ 测试，再翻译实现代码
5. **unsafe 最小化**：仅限 FFI 边界和零拷贝热路径，目标 < 10%

### 3.2 Sans-I/O 架构详解

传统网络库将协议解析与 socket I/O 耦合。Sans-I/O 设计将其分离：

```
┌─────────────────────────────────────────────────┐
│                   Application                     │
├─────────────────────────────────────────────────┤
│           zmq-context (Public API)                │
│     ZContext / ZSocket / Options / Monitor        │
├─────────────────────────────────────────────────┤
│           zmq-runtime (I/O Runtime)               │
│     Reactor / Poller / IO Threads / Signaler      │
├──────────────────────┬──────────────────────────┤
│    zmq-transport      │     zmq-core              │
│  (I/O Bound Layer)    │  (Pure Protocol Layer)     │
│                       │                            │
│  • TCP connect/bind   │  • ZMTP Codec              │
│  • IPC connect/bind   │  • Greeting/Handshake      │
│  • inproc channels    │  • Command (READY, SUB)    │
│  • Endpoint parsing   │  • Encoder/Decoder         │
│  • Async read/write   │  • Security Mechanisms     │
```

#### 1.1 yqueue — 无锁批量分配队列

**原文件**：`code/libzmq/src/yqueue.hpp`（188 行）

**C++ 核心设计**：
```cpp
template <typename T, int N> class yqueue_t {
    // 批量分配 N 个节点的块（chunk），用链表串联
    // begin_pos / end_pos / back_pos 三个原子指针
    // SPSC（单生产者单消费者）无锁设计
    struct chunk_t {
        T values[N];
        chunk_t *prev;
        chunk_t *next;
    };
    atomic_ptr_t<chunk_t> begin_chunk;
    atomic_ptr_t<chunk_t> back_chunk;
    chunk_t *end_chunk;  // 仅生产者可访问
};
```

**Rust 实现方案**：
```rust
// zmq-core/src/data_structures/yqueue.rs
use std::sync::atomic::{AtomicPtr, Ordering};
use std::ptr::NonNull;

const N: usize = 256; // 默认 chunk 大小

struct Chunk<T, const N: usize> {
    values: [MaybeUninit<T>; N],
    prev: Option<NonNull<Chunk<T, N>>>,
    next: Option<NonNull<Chunk<T, N>>>,
}

pub struct YQueue<T, const N: usize = 256> {
    begin_chunk: AtomicPtr<Chunk<T, N>>,
    back_chunk: AtomicPtr<Chunk<T, N>>,
    end_chunk: *mut Chunk<T, N>, // 仅生产者访问，无需原子
    begin_pos: usize,
    back_pos: usize,
    end_pos: usize,
    _marker: PhantomData<T>,
}
```

**unsafe 预期**：< 5%（仅限于 AtomicPtr 操作和 MaybeUninit 初始化）

**对应测试**：翻译 `unittests/unittest_ypipe.cpp`（70 行）→ `zmq-core/src/data_structures/yqueue.rs` 中的 `#[cfg(test)]` 模块

#### 1.2 ypipe — 无锁 SPSC 管道

**原文件**：`code/libzmq/src/ypipe.hpp`（120 行）

**C++ 核心设计**：
```cpp
template <typename T, int N> class ypipe_t {
    // 基于 yqueue 构建
    // write() → 写入消息
    // flush() → 原子地发布已写入的消息
    // check_read() → 检查是否有数据可读
    // read() → 读取一条消息
    // 支持乐观读取（probe）以提高性能
    yqueue_t<T, N> queue;
    T *w;       // 指向下一个写入位置
    T *r;       // 指向下一个读取位置
    bool flush; // 是否有待刷新数据
};
```

**Rust 实现方案**：基于 YQueue，提供更安全的类型化 SPI
```rust
pub struct YPipe<T, const N: usize = 256> {
    queue: YQueue<T, N>,
    w: *mut T,       // 仅生产者线程访问
    r: *mut T,       // 仅消费者线程访问
    flushed: AtomicBool,
    _marker: PhantomData<T>,
}

impl<T, const N: usize> YPipe<T, N> {
    pub fn write(&mut self, value: T) -> bool;
    pub fn flush(&mut self) -> bool;
    pub fn check_read(&self) -> bool;
    pub fn read(&mut self) -> Option<T>;
    pub fn probe<F, R>(&self, f: F) -> Option<R> where F: FnOnce(&T) -> R;
}
```

#### 1.5 ZmqMessage — 消息封装

**原文件**：`code/libzmq/src/msg.hpp` + `msg.cpp`（~200 行）

**C++ 核心设计**：
```cpp
class msg_t {
    // 64 字节固定大小（缓存行友好）
    // VSM（Very Small Message）：≤30 字节，数据内联在 msg_t 中，零堆分配
    // Large message：堆分配 buffer，引用计数（msg_content_t）
    // 支持零拷贝（用户提供 buffer + 释放回调）
    unsigned char u;              // 类型标志位
```

```rust
}

impl ZmqMessage {

    pub fn data(&self) -> &[u8];
```

**Rust 实现方案**：参考 omq.rs 的 Patricia-trie（O(M) 而非 O(N×M)）
```rust
pub struct SubscriptionTrie {
    root: TrieNode,
    count: usize,
}

struct TrieNode {
    children: HashMap<u8, TrieNode>,
```

### 第 2 阶段：ZMTP 协议编码/解码（预计 3-4 天）

**目标**：实现与 I/O 完全无关的 ZMTP 3.x 协议核心。

**设计决策**：编解码器接收 `&[u8]` 输入，产出结构化事件/命令枚举，不触碰任何 socket 或 async 代码。

#### 2.1 ZMTP 协议概述

ZMTP（ZeroMQ Message Transport Protocol）定义了 ZeroMQ 的线协议：

```
连接建立流程：
  Client                          Server
    |                               |
    |--- Greeting (64 bytes) ------>|
    |<-- Greeting (64 bytes) -------|
    |--- READY command ------------->|
    |<-- READY command -------------|
    |--- Message / Command -------->|
    |<-- Message / Command ---------|
    |           ...                  |
```

**Greeting 格式（64 字节）**：
```
Byte 0:    0xFF (signature)
Byte 1-8:  0x00 × 8 (reserved)
Byte 9:    0x7F (final short)
Byte 10:   revision (0x03 for ZMTP 3.0)
Byte 11:   0x00 for NULL / 0x01 for PLAIN / 0x02 for CURVE
Byte 12-31: 0x00 (reserved)
Byte 32-51: Server-Identity (20 bytes)
Byte 52-63: 0x00 (reserved)
```

#### 2.2 编解码模块设计

**文件结构**：
```
zmq-core/src/codec/
├── mod.rs          # 公共导出：ZmqDecoder, ZmqEncoder, ZmqEvent, ZmqCommand
├── greeting.rs     # 握手包头：Greeting 结构体 + encode/decode
├── command.rs      # ZMTP 命令帧：Command 枚举 + encode/decode
├── framing.rs      # 帧格式：长帧 (≥255B) / 短帧 (<255B) + MORE 标志位
├── encoder.rs      # 编码器状态机：ZmqCommand → Vec<u8>
└── decoder.rs      # 解码器状态机：&[u8] → Vec<ZmqEvent>
```

```rust
/// 传输端点枚举
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum Endpoint {
    Tcp { host: String, port: u16, is_wildcard: bool },
    Ipc { path: PathBuf },
    Inproc { name: String },
}

impl FromStr for Endpoint {
    type Err = ZmqError;
    fn from_str(s: &str) -> Result<Self, Self::Err> {
        if let Some(rest) = s.strip_prefix("tcp://") {
            parse_tcp_endpoint(rest)
        } else if let Some(rest) = s.strip_prefix("ipc://") {
            parse_ipc_endpoint(rest)
        } else if let Some(rest) = s.strip_prefix("inproc://") {
            Ok(Endpoint::Inproc { name: rest.to_string() })
        } else {
            Err(ZmqError::InvalidEndpoint(s.to_string()))
        }
    }
}

impl fmt::Display for Endpoint {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            Endpoint::Tcp { host, port, .. } => write!(f, "tcp://{}:{}", host, port),
            Endpoint::Ipc { path } => write!(f, "ipc://{}", path.display()),
            Endpoint::Inproc { name } => write!(f, "inproc://{}", name),
        }
    }
}
```

```rust
// zmq-core/src/socket/req.rs

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum ReqState {
    /// 等待发送请求
    ReadyToSend,
    /// 请求已发送，等待回复
    WaitingForReply,
    /// 请求已发送，收到部分回复
    ReceivingReply,
}

pub struct ReqSocket {
    dealer: DealerSocket,
    state: ReqState,
    /// 当前请求的消息副本（用于重试）
    pending_request: Option<ZmqMessage>,
    /// 请求超时
    timeout: Option<Duration>,
    /// 重试次数
    retries: u32,
}

impl Socket for ReqSocket {
    fn xsend(&mut self, msg: ZmqMessage) -> Result<(), ZmqError> {
        if self.state != ReqState::ReadyToSend {
            return Err(ZmqError::InvalidState("REQ must send-recv-send-recv"));
        }
        // 添加分隔帧（REQ 协议要求）
        let mut request = msg;
        request.push_front(Bytes::new()); // 空分隔帧
        self.dealer.xsend(request)?;
        self.pending_request = Some(request);
        self.state = ReqState::WaitingForReply;
        Ok(())
    }

    fn xrecv(&mut self) -> Result<ZmqMessage, ZmqError> {
        if self.state == ReqState::ReadyToSend {
            return Err(ZmqError::InvalidState("REQ: must send before recv"));
        }
        let msg = self.dealer.xrecv()?;
        // 跳过前导分隔帧
        // 验证回复完整性
        if !msg.more() || msg.is_last() {
            self.state = ReqState::ReadyToSend;
        }
        Ok(msg)
    }
    // ...
}
```

#### REP socket 路由逻辑

```rust
// zmq-core/src/socket/rep.rs

pub struct RepSocket {
    router: RouterSocket,
    /// 当前正在处理的请求的 routing_id
    current_request_routing_id: Option<u32>,
    state: RepState,
}

enum RepState {
    /// 等待接收请求
    WaitingForRequest,
    /// 已接收请求，等待发送回复
    ReadyToReply { routing_id: u32 },
}

impl RepSocket {
    fn xrecv(&mut self) -> Result<ZmqMessage, ZmqError> {
        if self.state != RepState::WaitingForRequest {
            return Err(ZmqError::InvalidState("REP: must recv before send"));
        }
        let msg = self.router.xrecv()?;
        // 提取 routing_id 用于回复路由
        if let Some(id) = msg.routing_id() {
            self.state = RepState::ReadyToReply { routing_id: id };
        }
        Ok(msg)
    }

    fn xsend(&mut self, msg: ZmqMessage) -> Result<(), ZmqError> {
        if let RepState::ReadyToReply { routing_id } = self.state {
            let mut reply = msg;
            reply.set_routing_id(routing_id);
            self.router.xsend(reply)?;
            self.state = RepState::WaitingForRequest;
            Ok(())
        } else {
            Err(ZmqError::InvalidState("REP: must recv before send"))
        }
    }
}
```

```rust
/// 安全机制 trait
pub trait Mechanism: Send + Sync {
    fn mechanism_type(&self) -> SecurityMechanism;
    /// 处理握手数据
    fn process_handshake(&mut self, data: &[u8]) -> Result<MechanismResult, ZmqError>;
    /// 生成握手输出
    fn next_handshake_output(&mut self) -> Option<Vec<u8>>;
    /// 握手是否完成
    fn is_handshake_complete(&self) -> bool;
}

pub enum MechanismResult {
    Success { user_id: Option<String> },
    Temporary(Vec<u8>),  // 需要更多握手数据
    Error(ZmqError),
}

// NULL 机制（无安全）
pub struct NullMechanism;

impl Mechanism for NullMechanism {
    fn mechanism_type(&self) -> SecurityMechanism { SecurityMechanism::Null }
    fn process_handshake(&mut self, _data: &[u8]) -> Result<MechanismResult, ZmqError> {
        // NULL 机制无需握手
        Ok(MechanismResult::Success { user_id: None })
    }
    fn is_handshake_complete(&self) -> bool { true }
}

// PLAIN 机制（用户名/密码）
pub struct PlainServer {
    zap_client: ZapClient,
    state: PlainState,
}

impl Mechanism for PlainServer {
    fn process_handshake(&mut self, data: &[u8]) -> Result<MechanismResult, ZmqError> {
        // 解析 PLAIN 握手消息
        // [username_len(1)] [username] [password_len(1)] [password]
        // → 发送 ZAP 请求验证 → 等待 ZAP 回复 → Success/Error
    }
}
```

```rust
        socket: &ZSocket,
        ipv6: bool,

        } else {
            "tcp://127.0.0.1:*"
        };
        socket.bind(addr)?;
        socket.last_endpoint()

        assert_eq!(received.data(), b"Hello");
        server.send(received, SendFlags::empty()).unwrap();
        let echoed = client.recv(RecvFlags::empty()).unwrap();
        assert_eq!(echoed.data(), b"Hello");
    }
```

| **合计估算** | **~3-4%** | 远低于 10% 要求 |

#### 9.3 SAFETY 注释规范

每个 unsafe 块必须附带注释说明：
```rust
// SAFETY: begin_chunk 仅在创建和销毁时写入，消费端仅读取，生产端不访问
// 这保证了 SPSC 场景下不存在数据竞争
let begin = unsafe { self.begin_chunk.load(Ordering::Acquire) };
```

| 产出物 | 说明 |
|--------|------|
| `cargo geiger` 报告 | unsafe 占比 < 10% |
| `cargo miri test` 结果 | 无未定义行为 |
| SAFETY 注释 | 所有 unsafe 块均已注释 |

## 第五章：编译自愈闭环详细设计

### 5.1 机制概述

这是竞赛要求 #2 的核心实现。当重构导致库 API 变更时，agent 自动根据编译器错误栈修复下游代码。

### 5.2 错误分类与修复策略

| Rust 错误码 | 含义 | 自动修复策略 |
|------------|------|-------------|
| `E0308` | 类型不匹配 | 检查 C++ 原类型 → 修正 Rust 类型注解或添加 `into()` 转换 |
| `E0599` | 方法未找到 | 检查 trait bound → 补充 trait 实现或添加 `use` 导入 |
| `E0382` | 所有权移动 | 添加 `.clone()` 或改为 `&T` 引用传递 |
| `E0502` | 可变/不可变借用冲突 | 重构为 `RefCell` 或分离借用作用域 |
| `E0609` | 访问不存在的字段 | 对照 C++ 类成员 → 补充 struct 字段 |
| `E0061` | 函数参数数量不匹配 | 对照 C++ 函数签名 → 修正参数 |
| `E0560` | 结构体构造缺少字段 | 对照 C++ 构造函数 → 补充字段或添加 `Default` |
| `E0432` | 找不到模块/导入 | 对照文件路径 → 修正 `mod` / `use` 声明 |
| `E0277` | trait bound 不满足 | 添加 `#[derive()]` 或手动实现 trait |
| `E0107` | 类型参数数量错误 | 对照 C++ 模板参数 → 修正泛型参数 |

### 5.3 实现架构

```rust
/// 编译自愈引擎
pub struct SelfHealingEngine {
    /// 当前项目的文件索引
    file_index: FileIndex,
    /// C++ 源码引用（用于对比）
    cpp_reference: CppReference,
    /// 修复历史（用于学习）
    fix_history: Vec<FixRecord>,
}

impl SelfHealingEngine {
    /// 主循环：编译 → 失败 → 解析错误 → 修复 → 循环
    pub async fn heal_loop(&mut self) -> Result<(), Vec<HealError>> {
        let mut remaining_fixes = Vec::new();
        loop {
            let output = self.run_cargo_build().await?;
            if output.success {
                return Ok(());
            }
            let errors = self.parse_rustc_errors(&output.stderr);
            if errors.is_empty() {
                return Err(remaining_fixes);
            }
            for error in errors {
                match self.apply_fix(&error) {
                    Ok(_) => remaining_fixes.push(error.into()),
                    Err(e) => log::warn!("Fix failed: {}", e),
                }
            }
        }
    }

    fn apply_fix(&self, error: &RustcError) -> Result<(), FixError> {
        match error.code {
            "E0308" => self.fix_type_mismatch(error),
            "E0599" => self.fix_method_not_found(error),
            "E0382" => self.fix_move_error(error),
            "E0502" => self.fix_borrow_conflict(error),
            "E0609" => self.fix_field_not_found(error),
            // ...
            _ => Err(FixError::UnknownErrorCode(error.code.clone())),
        }
    }
}
```

## 第六章：风险分析与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| C++ 模板难以等价翻译 | 中 | 中 | Rust trait + 泛型替代模板；macro_rules! 替代 C 宏；必要时保留 C++ 语义 |
| C++ 隐式类型转换丢失 | 高 | 低 | Rust 编译器会捕获所有隐式转换，逐一显式化 |

| 零拷贝消息开销 | 中 | 高 | `Bytes` 引用计数 + 自定义 VSM ≤30B 栈分配 |
| 多线程死锁 | 低 | 高 | Rust 所有权系统天然防死锁；`loom` 测试并发正确性 |
| 50k 行代码量过大 | 中 | 中 | 渐进式迁移，每阶段产出可运行代码，Phase 1-9 逐步推进 |
| 第三方包不兼容 | 低 | 低 | 优先使用成熟 crate（tokio, mio, bytes, crossbeam） |
