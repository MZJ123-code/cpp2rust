# 进展日志

## 2026-06-11 — REQ/REP socket 完整实现

### REQ socket (`zmq-core/src/socket/req.rs`)

从桩重写为完整的请求-回复状态机，1:1 对齐 C++ `req.cpp`/`req.hpp`：

- **状态机**：`Ready → WaitingReply → Ready`，`message_begin` 标志跟踪多帧消息边界
- **Strict 模式**（默认）：等待回复期间 `xsend` 返回 `InvalidState`
- **Relaxed 模式**（`strict=false`）：新 send 自动重置状态，对应 C++ `ZMQ_REQ_RELAXED`
- **Correlation ID**（`correlate=true`）：4 字节单调递增请求 ID，回复必须匹配；不匹配的回复自动跳过
- **LoadBalancer**：发送时通过 LB round-robin 选择对等端
- **Reply pipe tracking**：回复仅从发送请求的同一 pipe 读取，其他 pipe 的消息被丢弃
- **Stale message draining**：每次新请求发送前清空所有 pipe 的积压消息
- **Pipe termination**：回复 pipe 终止时自动重置为 Ready 状态，支持重连后重传
- **17 个单元测试**：覆盖状态机转换、strict/relaxed 模式、correlation ID 验证/跳过、多帧消息、pipe 生命周期

### REP socket (`zmq-core/src/socket/rep.rs`)

从桩重写为完整的回复端状态机，1:1 对齐 C++ `rep.cpp`/`rep.hpp`：

- **状态机**：`ReceivingRequest → SendingReply → ReceivingRequest`
- **FairQueue**：通过 FQ round-robin 从所有连接的对等端公平接收请求
- **路由前缀保存**：接收时将路由帧（backtrace stack）复制到 `routing_prefix`，回复时自动写回
- **单对等端跟踪**：通过 `current_peer` 确保路由帧和回复体从/发往同一 pipe
- **多帧请求/回复**：完整支持 `more` 标志的多帧消息
- **18 个单元测试**：覆盖状态机转换、路由前缀保存/回写、公平队列轮询、多帧消息、pipe 终止重置

### 测试结果

- **35 测试全部通过**（REQ 17 + REP 18）
- **zmq-core 总计 266 测试全部通过，0 失败**
- **全 workspace `cargo check` 通过，无新增警告**

### 设计决策

- REQ/REP 均为 Sans-I/O 实现（无异步依赖），仅通过 Pipe API 进行数据读写
- 测试使用单 pipe 模式：`pipe.write_to_socket()` 模拟数据到达，`pipe.read_from_socket()` 验证发送
- REP 的 `recv_from_fq` 在 pipe 无数据时自动 deactivate，避免无限循环

## 2026-06-10 — 初始 XPUB/XSUB 实现 + SubscriptionTrie 修复

### XPUB 实现 (`zmq-core/src/socket/xpub.rs`)

完整替换桩实现，1:1 对齐 C++ `xpub.cpp`：

- **字段对齐**：pipes, SubscriptionTrie（subscriptions + manual_subscriptions）, 待发送通知队列（pending_data/pending_flags/pending_pipes）
- **选项支持**：verbose_subs/verbose_unsubs/verboser, manual, nodrop (lossy), only_first_subscribe, send_last_pipe, welcome_msg
- **`xattach_pipe`**：subscribe_to_all 添加空前缀到 trie、发送欢迎消息、读取待处理订阅
- **`xread_activated`**：读取订阅/取消订阅消息，更新 trie，排队通知供 xrecv 返回
- **`xsend`**：通过 SubscriptionTrie 进行话题匹配，将消息发送到匹配的管道
- **`xrecv`**：返回排队中的订阅通知消息
- **`xpipe_terminated`**：从 trie 中移除管道，必要时将取消订阅消息排队
- **22 个单元测试**

### XSUB 实现 (`zmq-core/src/socket/xsub.rs`)

完整替换桩实现，1:1 对齐 C++ `xsub.cpp`：

- **字段对齐**：pipes, SubscriptionTrie（本地话题缓存）, 预取消息（poll 优化）
- **`xsend`**：识别订阅/取消订阅消息，更新本地 trie，通过广播发送到所有管道
- **`xrecv`**：通过公平队列从管道接收，按本地订阅 trie 过滤
- **`xhas_out`**：始终返回 true（订阅随时可发送）
- **`xattach_pipe`**：向新对等端转发缓存的订阅
- **19 个单元测试**

### Bug 修复：SubscriptionTrie::remove_by_value

`rm_value_recursive` 中存在的 bug：从节点移除值时，`subtree_prefix_count` 未传播到祖先节点。已修复：现在会返回移除数量并在递归回溯时调整祖先节点的计数。当前全部 199 个单元测试均通过。

- 修复了 `codec/decoder.rs` 中 `CommandName::Hello` 和 `CommandName::Hiccup` 的穷尽性匹配（该修复来自先前会话）

| MSVC link.exe 不工作 | 切 GNU toolchain `stable-x86_64-pc-windows-gnu` |
| workspace 依赖不能 optional | 移入具体 crate 的 Cargo.toml |
| YPipe terminator 未初始化 | `write()` 中 `push()` 后 `ptr::write(back_mut(), T::default())` |
| Heap Corruption (STATUS_HEAP_CORRUPTION) | 同上：Drop 时 `drop_in_place` 操作了未初始化内存 |
| `&T as *mut T` 不允许直接转 | `std::ptr::from_ref()` 先转 `*const T` 再 cast |
| C++ 测试语义对齐 | trie `add()` 返回前缀是否首次添加，非值是否重复 |
| Clean rebuild 有时必须 | 编译缓存导致旧对象链接，`cargo clean` 后正常 |
| Pipe 双向共用同一 SPSC | `new_pair()` 返回同一 Arc 的两个克隆，PAIR 双向有竞争风险 |
