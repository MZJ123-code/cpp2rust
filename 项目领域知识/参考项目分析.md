# 参考项目深度分析

## 概述

本项目参考了三个主要的 Rust ZeroMQ 实现：
1. **zmq.rs** - 官方纯 Rust 实现
2. **rzmq** - 最活跃的实现，io_uring 加速
3. **omq.rs** - 架构最先进，Sans-I/O 核心

## zmq.rs（官方纯 Rust 实现）

### 基本信息

- **仓库**：https://github.com/zeromq/zmq.rs
- **版本**：v0.6.0 (2026-05-04)
- **协议**：MIT
- **Stars**：~1,400
- **状态**：Beta，ZMTP 协议完整

### 架构特点

```
src/
├── backend.rs          — GenericSocketBackend：peer 管理、round-robin 发送
├── fair_queue.rs       — FairQueue：BinaryHeap + HashSet 公平接收
├── write_queue.rs      — WriteQueue：批量 128 条消息写入
├── pub.rs / sub.rs     — PUB/SUB backend + fanout queue
├── req.rs / rep.rs     — REQ/REP 状态机
├── dealer.rs / router.rs — DEALER/ROUTER 分拆发送/接收句柄
├── reconnect.rs        — 自动重连（指数退避：100ms → 30s）
├── codec/              — ZMTP 编解码
├── transport/           — TCP/IPC 传输层
├── endpoint/            — 端点解析
└── async_rt/            — 异步运行时抽象层
```

### 已实现功能

| 类别 | 实现 | 状态 |
|------|------|------|
| **Socket 类型** | PUB, SUB, REQ, REP, DEALER, ROUTER, PULL, PUSH, XPUB, XSUB (11种) | ✅ |
| **传输** | TCP, IPC (Unix + Windows) | ✅ |
| **安全** | NULL 机制 | ✅ |
| **异步运行时** | Tokio / async-std / async-dispatcher (编译期选择) | ✅ |

### 设计亮点

1. **GenericSocketBackend**：统一的 socket 后端抽象
2. **FairQueue**：基于 BinaryHeap 的公平队列
3. **WriteQueue**：批量写入优化（128 条消息）
4. **多运行时支持**：编译期选择 Tokio/async-std

### 与本项目的对比

| 特性 | zmq.rs | 本项目 |
|------|--------|--------|
| **Sans-I/O** | 计划中 | ✅ 已实现 |
| **Socket 类型** | 11 | 19 |
| **安全机制** | NULL | NULL/PLAIN/CURVE |
| **运行时选择** | 编译期 | 固定 Tokio |
| **性能路线图** | 有 | 无 |

### 学习价值

- **GenericSocketBackend**：可参考其 peer 管理和 round-robin 发送
- **FairQueue**：BinaryHeap + HashSet 的公平调度实现
- **多运行时支持**：运行时抽象层的设计

## rzmq（最活跃实现，io_uring 加速）

### 基本信息

- **仓库**：https://github.com/excsn/rzmq
- **版本**：v0.5.18 (2026-06-09)
- **协议**：MPL-2.0
- **代码量**：~23,000 行 Rust
- **状态**：Beta，非常活跃（20 个版本/年）

### 架构特点

```
rzmq/core/src/
├── context.rs          — Context
├── error.rs            — ZmqError
├── message/            — Msg, Blob, Metadata
├── protocol/zmtp/      — ZMTP 协议细节
├── runtime/            — 核心异步运行时（邮箱、事件、命令）
├── security/           — NULL, PLAIN, CURVE, Noise_XX
├── sessionx/           — 连接会话（桥接 socket 与 engine）
├── socket/             — Socket types, options, events, patterns
│   ├── patterns/       — distributor, fair_queue, load_balancer, trie
│   └── core/           — 内部 socket 状态机
├── transport/          — TCP, IPC, inproc
├── throttle/           — 自适应 I/O 节流
└── io_uring_backend/   — Linux io_uring（feature-gated）
```

### 独特优势

1. **io_uring 加速**：
   - TCP Cork：批量发送，减少系统调用
   - Zero-copy send：零拷贝发送
   - Multishot receive：多 shot 接收

2. **自适应 I/O 节流**：
   - 公平引擎：概率性平衡入/出
   - 防止单一连接饿死其他连接

3. **现代安全**：
   - NULL + PLAIN + CURVE
   - Noise_XX：自定义加密协议

4. **ZMTP 3.1 完全合规**：
   - 可与 libzmq 互操作
   - 支持所有标准 socket 类型

### 性能数据

```
基准测试（Linux Kernel 6.x）：
- TCP 吞吐：超过 libzmq
- TCP 延迟：低于 libzmq
- io_uring + TCP Cork：显著性能提升
```

### 与本项目的对比

| 特性 | rzmq | 本项目 |
|------|------|--------|
| **io_uring** | ✅ | 无 |
| **Sans-I/O** | 无 | ✅ |
| **Socket 类型** | 8 | 19 |
| **安全机制** | NULL/PLAIN/CURVE/Noise | NULL/PLAIN/CURVE |
| **Windows** | 无 | 无 |
| **自适应节流** | ✅ | 无 |

### 学习价值

- **io_uring 集成**：可参考其 io_uring 后端实现
- **自适应节流**：公平引擎的设计思路
- **Noise_XX**：现代安全协议的实现

## omq.rs（架构最先进，Sans-I/O 核心）

### 基本信息

- **仓库**：https://github.com/Paddor/omq.rs
- **版本**：v0.5.3
- **协议**：ISC
- **状态**：Alpha，声称比 libzmq 快 3 倍

### 架构特点

```
├── omq-proto    — Sans-I/O ZMTP 3.x 核心（关键参考！）
├── omq-tokio    — Tokio 后端
├── omq-compio   — compio 后端（Linux io_uring / Windows IOCP）
├── omq-libzmq   — libzmq 兼容 C 接口
├── blume        — 批量 MPSC 通道
├── yring        — 有界 SPSC 环形缓冲区
└── pyomq        — Python 绑定（PyO3）
```

### 最完整实现

| 类别 | 实现 | 说明 |
|------|------|------|
| **Socket 类型** | 20 种 | 11 标准 + 9 草案 |
| **传输** | 8 种 | TCP, IPC, inproc, UDP, WS, WSS, lz4+tcp, zstd+tcp |
| **安全** | 3 种 | PLAIN, CURVE, BLAKE3ZMQ |
| **测试** | 750+ | 集成测试 + 协议模糊测试 + 12 浸泡测试 |

    }

### 性能优化

1. **Patricia-trie 订阅匹配**：O(M) 而非 O(N×M)
2. **工作窃取发送泵**：多线程负载均衡
3. **Per-socket HWM**：每 socket 高水位标记
4. **zstd 字典自动训练**：首个 1k 消息自动训练压缩字典

## 三个项目的共同模式

### 1. 异步运行时选择

| 项目 | 默认运行时 | 可选运行时 |
|------|-----------|-----------|
| zmq.rs | Tokio | async-std, async-dispatcher |
| rzmq | Tokio | 无 |
| omq.rs | Tokio | compio (io_uring/IOCP) |

**结论**：Tokio 是事实标准，本项目选择 Tokio 是正确的。

### 2. 消息模式实现

所有项目都实现了核心消息模式：
- PUB/SUB（发布/订阅）
- REQ/REP（请求/响应）
- PUSH/PULL（管道）
- DEALER/ROUTER（路由）

**差异**：
- zmq.rs：11 种 socket 类型
- rzmq：8 种 socket 类型
- omq.rs：20 种 socket 类型（最全）
- 本项目：19 种 socket 类型

### 3. 安全机制支持

| 项目 | NULL | PLAIN | CURVE | Noise | BLAKE3 |
|------|------|-------|-------|-------|--------|
| zmq.rs | ✅ | ❌ | ❌ | ❌ | ❌ |
| rzmq | ✅ | ✅ | ✅ | ✅ | ❌ |
| omq.rs | ❌ | ✅ | ✅ | ❌ | ✅ |
| 本项目 | ✅ | ✅ | ✅ | ❌ | ❌ |

**结论**：本项目的安全机制支持处于中上水平。
