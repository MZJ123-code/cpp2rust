# 设计决策记录

## 概述

本文档记录了 libzmq Rust 重写项目中的关键设计决策，包括决策背景、方案选择、权衡分析和最终结论。

## 决策 1：Sans-I/O 架构

### 背景

ZeroMQ 是一个高性能异步消息库，需要处理复杂的协议解析和 I/O 操作。传统实现将协议逻辑与 I/O 耦合，导致：

1. 测试困难（需要启动真实 socket）
2. 运行时绑定（难以切换异步运行时）
3. 性能优化受限（I/O 层成为瓶颈）

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **传统耦合架构** | 简单直接 | 测试困难、运行时绑定 |
| **Sans-I/O 架构** | 可测试、运行时无关、性能优化空间大 | 初期设计复杂度高 |
| **混合架构** | 平衡复杂度和灵活性 | 边界不清晰 |

### 决策

**选择 Sans-I/O 架构**

理由：
1. 可测试性是首要需求（竞赛要求 143 个测试 1:1 映射）
2. 运行时无关性允许未来切换到 io_uring 等新技术
3. 性能优化空间大（协议层可进行激进优化）

### 实现

```
zmq-core（Sans-I/O 核心）
  ├── 零 I/O 依赖
  ├── 纯函数接口
  └── 确定性行为

zmq-transport（I/O 绑定层）
  ├── 引用 zmq-core
  ├── 使用 Tokio
  └── 处理实际网络 I/O
```

### 验证

- 单元测试：纯函数测试，无需 I/O
- 性能测试：inproc 吞吐超过 C++ 121%
- 架构验证：模块依赖关系清晰

## 决策 2：异步运行时选择

### 背景

Rust 有多个异步运行时：
- Tokio：最成熟，生态最丰富
- async-std：标准库风格
- smol：轻量级
- compio：io_uring/IOCP 支持

### 方案对比

| 运行时 | 优点 | 缺点 |
|--------|------|------|
| **Tokio** | 生态最成熟、文档最全 | 较重 |
| **async-std** | 标准库风格 | 社区较小 |
| **smol** | 轻量级 | 功能有限 |
| **compio** | io_uring/IOCP 支持 | 较新 |

### 决策

**选择 Tokio 作为默认运行时**

理由：
1. 生态最成熟，文档最全
2. zmq.rs 和 rzmq 都选择 Tokio，可参考实现
3. 性能优秀，稳定性好
4. 与现有 Rust 异步库兼容性最好

### 实现

```toml
# zmq-transport/Cargo.toml
[dependencies]
tokio = { version = "1", features = ["full"] }
```

```rust
// 使用 Tokio 的异步接口
use tokio::net::{TcpListener, TcpStream};

pub async fn accept() -> Result<TcpStream> {
    let listener = TcpListener::bind("0.0.0.0:0").await?;
    let (stream, _) = listener.accept().await?;
    Ok(stream)
}
```

### 未来扩展

- 可以通过 feature flag 支持其他运行时
- 可以添加 compio 后端支持 io_uring/IOCP

## 决策 3：消息表示

### 背景

libzmq 的 `msg_t` 是一个 64 字节的固定大小结构，支持：
- VSM（Very Small Message）：≤30 字节内联存储
- 大消息：堆分配 + 引用计数
- 零拷贝：用户提供 buffer + 释放回调

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **直接翻译 msg_t** | 与 C++ 完全兼容 | unsafe 多、不符合 Rust 习惯 |
| **使用 bytes::Bytes** | 零拷贝、引用计数 | 不支持 VSM 优化 |
| **自定义 ZmqMessage** | 可以优化 VSM | 实现复杂 |

### 决策

**使用自定义 ZmqMessage + bytes::Bytes**

理由：
1. VSM 优化对性能至关重要（30B 消息占 ZeroMQ 使用的 80%+）
2. bytes::Bytes 提供零拷贝和引用计数
3. 可以在保持 Rust 安全性的同时实现 VSM

### 实现

```rust
pub struct ZmqMessage {
    frames: VecDeque<Bytes>,
    routing_id: Option<u32>,
    group: Option<String>,
    flags: MessageFlags,
}

enum MessagePayload {
    Vsm([u8; 30], u8),           // 栈上小消息（VSM）
    Shared(Bytes),                 // 引用计数堆消息
    ZeroCopy(Box<dyn Drop + Send>), // 用户提供的内存（零拷贝）
}

impl ZmqMessage {
    /// 创建小消息（VSM 优化）
    pub fn from_slice(data: &[u8]) -> Self {
        if data.len() <= 30 {
            // VSM：内联存储，零堆分配
            let mut buf = [0u8; 30];
            buf[..data.len()].copy_from_slice(data);
            ZmqMessage {
                frames: VecDeque::new(),
                routing_id: None,
                group: None,
                flags: MessageFlags::new(),
            }
        } else {
            // 大消息：堆分配 + 引用计数
            ZmqMessage {
                frames: VecDeque::from([Bytes::copy_from_slice(data)]),
                routing_id: None,
                group: None,
                flags: MessageFlags::new(),
            }
        }
    }
}
```

### 验证

- VSM 消息：0 堆分配（性能测试验证）
- 大消息：bytes::Bytes 引用计数（正确性测试验证）
- 零拷贝：用户 buffer 直接使用（集成测试验证）

## 决策 4：无锁数据结构

### 背景

libzmq 使用多个无锁数据结构：
- `yqueue_t`：批量分配队列
- `ypipe_t`：SPSC 无锁管道
- `array_t`：O(1) 删除索引数组

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **直接翻译 C++ 代码** | 与 C++ 行为一致 | unsafe 多、难以验证正确性 |
| **使用标准库并发原语** | 安全、易验证 | 性能可能不如无锁 |
| **使用 crossbeam** | 高性能、安全 | 依赖外部 crate |

### 决策

**混合策略：关键路径使用 unsafe 无锁，其他使用标准库**

理由：
1. 无锁数据结构是性能关键（SPSC 管道用于 socket 间通信）
2. Rust 的 unsafe 可以控制，每处都需要 `// SAFETY:` 注释
3. 非关键路径使用标准库更安全

### 实现

```rust
// zmq-core/src/data_structures/yqueue.rs
// 关键路径：使用 unsafe 实现无锁队列

struct Chunk<T, const N: usize> {
    values: [MaybeUninit<T>; N],
    prev: Option<NonNull<Chunk<T, N>>>,
    next: Option<NonNull<Chunk<T, N>>>,
}

pub struct YQueue<T, const N: usize = 256> {
    begin_chunk: AtomicPtr<Chunk<T, N>>,
    back_chunk: AtomicPtr<Chunk<T, N>>,
    end_chunk: *mut Chunk<T, N>, // 仅生产者访问
    // ...
}

impl<T, const N: usize> YQueue<T, N> {
    /// 写入元素
    /// SAFETY: 仅生产者线程调用
    pub unsafe fn write(&mut self, value: T) -> bool {
        // 无锁写入实现
    }
}
```

```rust
// zmq-core/src/data_structures/array.rs
// 非关键路径：使用标准库

pub struct IndexArray<T> {
    items: Vec<T>,
    index_map: HashMap<usize, usize>,
}

impl<T> IndexArray<T> {
    pub fn push(&mut self, item: T) -> usize {
        // 使用标准库实现
    }

    pub fn remove(&mut self, key: usize) -> Option<T> {
        // O(1) swap_remove
    }
}
```

### 验证

- unsafe 代码：使用 Miri 测试未定义行为
- 并发安全：使用 ThreadSanitizer 检测数据竞争
- 性能测试：验证无锁实现的性能优势

## 决策 5：错误处理策略

### 背景

libzmq 使用 `errno` 风格的错误处理，Rust 有更丰富的错误处理模式。

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **直接翻译 errno** | 与 C++ 兼容 | 不符合 Rust 习惯 |
| **使用 anyhow** | 简单易用 | 错误类型不明确 |
| **自定义错误枚举** | 类型安全、可恢复 | 实现复杂 |

### 决策

**自定义错误枚举 + thiserror**

理由：
1. 类型安全：编译时检查错误处理
2. 可恢复：不同错误类型可以不同处理
3. 与 Rust 生态兼容：可转换为 anyhow

### 实现

```rust
use thiserror::Error;

#[derive(Error, Debug)]
pub enum ZmqError {
    #[error("codec error: {0}")]
    Codec(#[from] CodecError),

    #[error("protocol error: {0}")]
    Protocol(#[from] ProtocolError),

    #[error("security error: {0}")]
    Security(#[from] SecurityError),

    #[error("io error: {0}")]
    Io(#[from] std::io::Error),
}

#[derive(Error, Debug)]
pub enum CodecError {
    #[error("invalid greeting signature")]
    InvalidGreetingSignature,

    #[error("unsupported ZMTP version: {0}")]
    UnsupportedVersion(u8),

    #[error("invalid frame size: {0}")]
    InvalidFrameSize(usize),
}

#[derive(Error, Debug)]
pub enum ProtocolError {
    #[error("missing required command: {0}")]
    MissingCommand(String),

    #[error("invalid state transition")]
    InvalidStateTransition,
}
```

### 验证

- 编译时检查：所有错误都必须处理
- 错误恢复：不同错误类型可以不同处理
- 与 anyhow 兼容：可转换为动态错误

## 决策 6：测试策略

### 背景

竞赛要求：
1. `code/libzmq/tests` 全部转成 Rust 并测试通过
2. 性能基线对比 C++ 版本不小于 95%
3. unsafe 比例小于 10%

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **仅集成测试** | 简单直接 | 覆盖不全面 |
| **单元测试 + 集成测试** | 覆盖全面 | 工作量大 |
| **单元 + 集成 + 模糊测试** | 最全面 | 工作量最大 |

### 决策

**单元测试 + 集成测试 + Miri 测试**

理由：
1. 单元测试：验证数据结构和算法的正确性
2. 集成测试：验证模块间的交互
3. Miri 测试：验证 unsafe 代码的正确性

### 实现

```rust
// 单元测试（内联在源码中）
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_ypipe_write_read() {
        let mut pipe = YPipe::new();
        assert!(pipe.write(42));
        assert!(pipe.flush());
        assert!(pipe.check_read());
        assert_eq!(pipe.read(), Some(42));
    }
}

// 集成测试（tests/ 目录）
#[test]
fn test_pub_sub() {
    let ctx = ZContext::new();
    let pub = ctx.socket(SocketType::PUB).unwrap();
    let sub = ctx.socket(SocketType::SUB).unwrap();

    pub.bind("inproc://test").unwrap();
    sub.connect("inproc://test").unwrap();

    // 发送和接收消息
}
```

### 验证

- 测试覆盖：164 个测试通过
- Miri 测试：unsafe 代码无未定义行为
- 性能测试：达到 C++ 95% 以上

## 总结

### 决策原则

1. **性能优先**：关键路径使用无锁数据结构
2. **安全第一**：unsafe 代码最小化，每处都有注释
3. **可测试性**：Sans-I/O 架构允许纯函数测试
4. **Rust 惯用**：使用 Rust 的错误处理和类型系统

### 决策影响

| 决策 | 正面影响 | 负面影响 |
|------|---------|---------|
| **Sans-I/O** | 可测试、运行时无关 | 初期设计复杂 |
| **Tokio** | 生态成熟、文档全 | 较重 |
| **自定义 ZmqMessage** | VSM 优化 | 实现复杂 |
| **无锁数据结构** | 高性能 | unsafe 多 |
| **自定义错误枚举** | 类型安全 | 实现复杂 |
| **单元 + 集成测试** | 覆盖全面 | 工作量大 |

### 未来决策点

1. **io_uring 支持**：是否在 zmq-transport 中添加？
2. **其他运行时**：是否支持 async-std/smol？
3. **更多 socket 类型**：是否实现所有 20 种？

这些决策将根据项目进展和需求变化进行调整。
